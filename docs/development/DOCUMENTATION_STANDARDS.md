# Documentation Standards

- 文档描述当前真实状态，不把计划写成已实现；使用 `Planned`、`Proposed`、`Not implemented` 标识未来内容。
- 职责、公共接口、配置、运行方式、输入输出、依赖、数据格式、限制或测试方法变化时同步对应文档。
- 仅内部实现变化且外部行为不变时，可只追加编辑日志，避免无意义修改。
- 每个正式模块都有 `docs/modules/<module-name>.md`；模块映射与代码保持一致。
- 长期结构决策使用 ADR；已接受 ADR 不静默改写。
- `PROJECT_STATE.md` 说明“现在是什么”，`EDIT_LOG.md` 说明“经历了什么”，`CHANGELOG.md` 说明用户可见或版本重要变化。
- `PROJECT_COMPASS.md` 说明稳定项目方向、当前语义、意图/假设/开放决定概要和 AI 自审；详细实现、历史和问题仍由各自文档负责。
- `BUG_LOG.md` 保存已发现错误与可信潜在缺陷的完整历史；`KNOWN_ISSUES.md` 只保存当前仍影响用户的问题摘要，二者不得把计划功能伪装成Bug。
- 文档应简洁、可链接、避免重复权威来源；更新日期只能反映实际验证时间。
- 命令、检查和结果必须真实，不得记录未执行操作。
