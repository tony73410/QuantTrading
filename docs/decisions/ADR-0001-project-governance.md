# ADR-0001: Project Governance Foundation

## Status

Accepted

## Context

项目需要在交易功能出现前建立可维护、可追踪、可测试、可回滚的基础，同时保留用户对业务和技术选择的决定权。首次检查时工作目录为空且未初始化 Git。

## Options considered

1. 立即选择语言、框架和业务模块并搭建应用。
2. 只创建没有说明的占位目录。
3. 建立语言无关、内容明确的治理和目录骨架，后续由实际需求驱动技术与模块选择。

## Decision

采用选项 3。建立仓库级规则、文档索引、状态/决策/模块/开发标准、只追加编辑日志和目录职责说明；不初始化 Git，不选择技术栈，不实现交易功能。

## Rationale

该方案满足当前明确授权，并避免提前锁定技术、接口和业务边界。它为后续小步、可审查改动提供统一记录和审批机制。

## Consequences

- 优点：职责和审批边界明确，变化可追踪，未来模块由真实需求驱动。
- 限制：尚无可运行程序、自动化测试或技术栈专用工具。
- 要求：后续任务必须同步当前状态、相关文档和编辑日志。

## Reversal

若未来治理方式需要调整，应创建新的 ADR 说明替代方案、迁移影响和回滚路径，并将本 ADR 标记为 Superseded；不得抹去本决策历史。
